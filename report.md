## 1. Доработка сервиса и повышение надёжности

В сервис были добавлены проверки и обработка ошибок, чтобы приложение продолжало корректно работать при нештатных сценариях и возвращало понятные ответы клиенту.

Обработанные проблемные ситуации:

- сервис стартует без падения, даже если модель не загрузилась при запуске; в этом случае `/health` возвращает `degraded`, а не аварийное завершение процесса;
- запрос в `/predict` при отсутствии загруженной модели обрабатывается корректно и возвращает `503 Service Unavailable`;
- в `/updateModel` обрабатывается невалидный или пустой `run_id`;
- неудачное обновление модели не затирает текущую рабочую модель;
- отсутствующие обязательные фичи для текущей модели приводят к корректной ошибке `422`;
- лишние поля во входном JSON запрещены на уровне схемы запроса;
- отрицательные значения для числовых признаков, где они не ожидаются, отсекаются валидацией;
- модель без `feature_names_in_` считается некорректной и не активируется;
- модель без `predict_proba` считается некорректной и не активируется;
- некорректный результат инференса, например неожидаемая форма вероятностей или нечисловое значение, обрабатывается как ошибка сервиса;
- при смене модели сбрасывается буфер накопленных данных для Evidently, чтобы не смешивать статистику разных моделей.

Дополнительно был доработан `/health`: он возвращает текущий `run_id`, тип модели, список требуемых фичей и текст последней ошибки, если модель недоступна.

Пример ответа `/health` после успешной загрузки модели:

```json
{
  "status": "ok",
  "run_id": "edfad2bc1f1e4681a4174ee5bb09bd35",
  "model_type": "RandomForestClassifier",
  "features": [
    "race",
    "sex",
    "native.country",
    "occupation",
    "education",
    "capital.gain"
  ],
  "error": null
}
```

## 2. Тесты

На сервис написаны unit- и integration-тесты. Тестовый набор покрывает больше трёх критериев из задания.

Что проверяется:

- функции предобработки данных;
- успешная загрузка модели и корректный инференс;
- поведение контейнера модели при ошибках загрузки;
- логика хэндлеров `/predict`, `/updateModel`, `/health`;
- валидация входных запросов;
- работа сервиса целиком через `TestClient`;
- логика накопления и отправки drift-отчётов через Evidently.

Добавленные тесты:

- проверка выбора нужных колонок и порядка признаков при подготовке входа;
- проверка ошибки при отсутствии обязательных фичей;
- проверка ошибки при несовместимой схеме признаков;
- проверка успешной загрузки модели и сохранения метаданных;
- проверка того, что неуспешное обновление модели не затирает рабочую версию;
- проверка успешного `predict` и проверки формы `predict_proba`;
- проверка поведения при падении `predict_proba`;
- проверка реакции API на отсутствие модели;
- проверка `422` на невалидные payload-запросы;
- smoke-тест сервиса с успешным запросом на `/predict`;
- проверка экспорта пользовательских Prometheus-метрик через `/metrics`;
- проверка накопления reference/current буферов для Evidently и отправки отчёта.

Тесты запускались командой:

```bash
python3 -m unittest discover -s tests -v
```

В рамках разработки дополнительно использовалась быстрая проверка импорта и синтаксиса:

```bash
python3 -m compileall ml_service tests main.py
```

## 3. Мониторинг сервиса

В сервис добавлен Prometheus-совместимый эндпоинт `/metrics`. Все пользовательские метрики имеют префикс `bor_`, чтобы не пересекаться с метриками других студентов в общем Prometheus.

Пример экспортируемых метрик:

```text
bor_ml_service_requests_total
bor_ml_service_request_duration_seconds
bor_ml_service_preprocess_duration_seconds
bor_ml_service_inference_duration_seconds
bor_ml_service_model_probability
bor_ml_service_current_model_info
```

### 3.1. Технические метрики

Логируются:

- общее число запросов;
- распределение запросов по HTTP status code;
- время ответа сервиса;
- число запросов, которые находятся в обработке;
- системные метрики Python/процесса, экспортируемые `prometheus_client` по умолчанию.

Примеры визуализаций в Grafana:

- `Request Rate`;
- `HTTP Status Codes`;
- `Request Latency P75/P90/P95/P99/P99.9`;
- `In-Progress Requests`.

### 3.2. Метрики, связанные с входными данными

Логируются:

- время предобработки входного запроса;
- распределение числовых признаков;
- частоты категориальных значений признаков.

Примеры визуализаций в Grafana:

- `Preprocess Latency P75/P90/P95/P99/P99.9`;
- `Capital Gain Distribution`;
- `Race Distribution`;
- `Education Distribution`.

### 3.3. Метрики работы модели

Логируются:

- время инференса модели;
- распределение предсказанных классов;
- распределение вероятностей положительного класса.

Примеры визуализаций в Grafana:

- `Inference Latency P75/P90/P95/P99/P99.9`;
- `Prediction Distribution`;
- `Probability Distribution`.

### 3.4. Метрики по текущей модели и её обновлению

Логируются:

- признак доступности текущей модели;
- число признаков, необходимых текущей модели;
- тип текущей модели;
- `run_id` текущей модели;
- список фичей текущей модели;
- число успешных и неуспешных обновлений модели;
- время последнего успешного обновления.

Примеры визуализаций в Grafana:

- `Current Model Available`;
- `Current Model Feature Count`;
- `Current Model Info`;
- `Model Update Attempts`.

Для временных метрик в Grafana использовались перцентили `75`, `90`, `95`, `99`, `99.9`, рассчитанные через `histogram_quantile`.

Пример запроса для панели с `p95` времени ответа сервиса:

```promql
histogram_quantile(
  0.95,
  sum by (le) (rate(bor_ml_service_request_duration_seconds_bucket[5m]))
)
```

## 4. Алертинг

В Grafana были добавлены алерты на техническое состояние сервиса и на поведение модели. Для уведомлений использован Telegram contact point, чтобы проверяющий мог увидеть реальную доставку уведомлений.

Добавленные алерты:

- `BOR Service Unavailable`;
- `BOR High Request Latency P95`;
- `BOR High Inference Latency P95`;
- `BOR 5xx Responses`;
- `BOR Prediction Drift To Class 1`;
- `BOR Probability P95 Too High`.

При настройке правил для метрик, у которых отсутствие данных допустимо, режим `No data` был переведён в `OK`, чтобы избежать ложных срабатываний вида `DatasourceNoData`.

Для демонстрации срабатывания алертов использовались следующие сценарии:

- запуск сервиса с невалидным `run_id`, чтобы показать недоступность модели и `5xx`;
- генерация смещённого потока запросов, чтобы показать сдвиг распределения предсказаний и вероятностей;
- нагрузочный сценарий для проверки алертов на latency.

Пример выражения для алерта на `5xx`:

```promql
sum(increase(bor_ml_service_requests_total{status_code=~"5.."}[5m])) > 0
```

## 5. Мониторинг дрифта через Evidently

В сервис добавлен фоновый мониторинг дрифта данных и предсказаний с использованием Evidently.

Реализованный подход:

- после каждого успешного запроса в памяти сервиса сохраняются входные признаки, `prediction` и `probability`;
- первые `N` запросов формируют reference-выборку;
- следующие `N` запросов формируют current-выборку;
- фоновая корутина с заданным интервалом пытается построить отчёт `DataDriftPreset`;
- после успешной отправки в Evidently current-буфер очищается и начинается накопление следующего окна;
- при смене модели накопленные буферы сбрасываются.

Использованные параметры для демонстрации:

- `EVIDENTLY_URL=http://158.160.2.37:8000/`;
- `EVIDENTLY_REPORT_INTERVAL_SECONDS=30`;
- `EVIDENTLY_REFERENCE_SIZE=20`;
- `EVIDENTLY_CURRENT_SIZE=20`.

Пример запуска контейнера для демонстрации drift-мониторинга:

```bash
docker run --rm \
  -p 8890:8890 \
  -e MLFLOW_TRACKING_URI=http://158.160.2.37:5000/ \
  -e DEFAULT_RUN_ID=edfad2bc1f1e4681a4174ee5bb09bd35 \
  -e EVIDENTLY_URL=http://158.160.2.37:8000/ \
  -e EVIDENTLY_PROJECT_ID=<project_id> \
  -e EVIDENTLY_REPORT_INTERVAL_SECONDS=30 \
  -e EVIDENTLY_REFERENCE_SIZE=20 \
  -e EVIDENTLY_CURRENT_SIZE=20 \
  --name bor-ml-service \
  bor-ml-service
```

В Evidently были получены отчёты как для базового потока запросов, так и для смещённого трафика. Факт drift был зафиксирован отдельным запуском после генерации целенаправленно изменённого потока входных данных.
